## 1. 투고 가능성 요소 프레임워크 (측정 가능 지표)

| 요소 | 정의 | 측정 지표 | 현재 측정값 (원고 공통) |
|---|---|---|---|
| F1 범위 적합성 | venue 규정이 해당 유형을 명시 | 2=규정 원문 확인 / 1=허용 추정·문의 필요 / 0=범위 밖 | venue별 |
| F2 기여 신선함 | 차별점 수 (역사→이론→공개랩 통합 서사, 유도 완전 분해, 한국어 접근성, 테스트된 교육 코드) | 해당 venue에 유효한 차별점 개수(0~4) | venue별 |
| F3 신뢰도 | 기계 검증 가능한 품질 신호 | tests 340+760 pass, coverage 84%(floor 80 CI), formula validator 0 fail, 앵커 113, CI green | **측정 완료, 전 유닛 공통 강점** |
| F4 타당성 | 주장-근거 결합도 | 인용 커버리지 29/29(100%), 수치 예시 기계 검산 7종 / 학습효과 실측 데이터 **0건** | 인용 100% / 학습 데이터 0 |
| F5 분량 실행 가능성 | 필요 압축률·신규 작성량 → 작업 주 수 | 압축률 %, 신규 작성 쪽, 추정 주 수 | venue별 |
| F6 언어 비용 | 영문화 필요 분량 | 번역/재작성 쪽 수 | venue별 |
| F7 금전 비용 | APC/게재료/등록비 | 원화 환산 | venue별 |
| F8 일정 여유 | slack = (마감까지 주 수) − (필요 주 수) | 주 (rolling이면 목표일 기준) | venue별 |
| F9 정책 리스크 | 미확인 규정·자격 요건 수 | 문의/확인 필요 건수 | venue별 |
| F10 이중게재 방어 | 2차 게재/재사용 규정 충족 | 편집자 사전 서면 동의 여부(0/1) + 신규 콘텐츠 비율(%) | venue별 |
| F11 심사 생존성 | 투고 후 통과 가능성 | 해당 venue 최근 3년 유사(튜토리얼/리뷰/교육랩) 게재 사례 수 + 에디터 사전 문의 회신(0/1) | venue별, 투고 전 측정 |

분류 규칙:
- **A (즉시 실행)**: F1=2, slack≥4주, F9≤1(경미), 신규 데이터 불필요
- **B (작업 후 가능)**: 압축·번역·재구성 작업만 하면 투고 가능
- **C (갭 보완 필요)**: 데이터 수집 또는 정책 확인 결과에 종속
- **D (제외)**: 범위 밖, 중복 게재 충돌, 비효율

## 4. 중복 게재(이중 출판) 방지 설계 — 핵심 리스크

같은 내용을 국문/영문으로 두 곳에 내는 것은 이중 게재 규정 위반 소지가 있다.
회피 설계: **유닛별 내용 축을 분리**한다.

| 투고물 | 내용 축 | 겹침 관리 |
|---|---|---|
| JOSE (U1) | 소프트웨어·교육자료 자체 | 논문류와 성격 다름(코드 리뷰), 상호 인용 |
| ACC (U2) | 임피던스 "제어" 튜토리얼 (6장 핵심) | U5와 겹침 최대 지점 — U5에서 제어 파트를 요약하고 ACC판 인용 |
| KROS (U3) | 임피던스 "기초·역사" (2~5장, 한국어) | 영문 U5와 언어·심도 분리, 투고 시 선행 게재물 고지 |

관련 산출물 고지. 이 체크를 투고 게이트(아래 F10)로 추가한다.

## Lessons

- 레드팀 리뷰가 기획 단계에서 결격 사유(ACC 세션 구조)를 잡음 — venue "존재"
  확인과 venue "자격" 확인은 다른 검증이다. 이후 모든 venue는 저자 자격
  요건(회원, 소속, 세션 구조)을 F9에서 명시적으로 채점할 것.
- 기록 간 일관성: VENUE_SEARCH가 맞고 PLAN이 완화한 사례 — 파생 문서를 만들 때
  원 기록의 제약을 약화하지 않았는지 대조할 것.
