## 남은 3종 — 인터랙티브 뷰어/메뉴 (사용자 우선순위 결정 필요)

### A. 자동 반복 (`--loop`) — 난이도 中

- **원하는 동작(자연스러운 기본값)**: 뷰어가 sim_time에 도달하면 정지하지 말고 초기 상태로 리셋해 **창을 닫을 때까지 계속 재생**.
- **구현 가능성**: 4개 랩이 공통 훅 `ExperimentResetControl` + `reset_control.consume()`를 이미 가짐(리셋 버튼용). sim_time 도달 시 이 리셋을 자동 트리거하면 됨.
- **주의점**: (1) 루프 조건 변경 시 로거가 매 루프 기록을 누적 → 저장 산출물 비대화. 해결책: 루프 모드는 **첫 패스만 저장하고 이후엔 시각적 재생만**. (2) 창닫힘 감지(`viewer.is_running`) 정확히. (3) 4개 파일 + 테스트.
- **결정 필요**: 무한 반복이 기본이면 되는지? 아니면 "N회 반복"도 옵션으로?

### B. 다시 누르면 새 창 (단일 인스턴스) — 난이도 中

- **원하는 동작(택1, 사용자 결정)**:
  - (a) 같은 랩을 다시 실행하면 **이전 뷰어 창을 닫고** 새로 열기, 또는
  - (b) 이미 열린 창이 있으면 **그 창을 앞으로 가져오기**(재실행 무시), 또는
  - (c) 지금처럼 여러 창 허용하되 메뉴에 "열린 창 닫기" 버튼 추가.
- **구현**: 메뉴(`learner_menu.py`)가 실행 서브프로세스 핸들을 추적 → 재실행 전 이전 프로세스 종료. (a)가 가장 단순하고 직관적.

### C. 버튼 딜레이 — 난이도 中

- **원인**: 버튼 1회가 풀 파이프라인(시뮬+플롯+리포트+브라우저). 특히 플롯·리포트 생성이 무겁고, 브라우저 자동 열기가 체감 지연.
- **개선안(조합 가능)**:
  - (1) **빠른 미리보기 모드**: 뷰어만 열고 플롯·리포트 생성을 건너뛰는 `--quick`(또는 메뉴의 "빠른 실행" 버튼). 관찰만 할 땐 즉시.
  - (2) 리포트·브라우저 열기를 **백그라운드**로 (시뮬 종료 후 비동기).
  - (3) 첫 클릭 시 MuJoCo 프리로드(모델 캐시)로 2회차부터 빠르게.
- **결정 필요**: 기본을 "빠른 실행(그래프 없음)"으로 할지, 아니면 지금처럼 풀 파이프라인을 기본으로 두고 빠른 모드를 옵션으로 둘지.

## 권고 우선순위

1. **C-(1) 빠른 실행 모드** — 체감 개선 가장 큼, 위험 낮음(옵션 추가). "그냥 움직이는 것만 빨리 보고 싶다"에 직접 응답.
2. **A 자동 반복** — 구체적 요청, 공통 훅 존재. 첫 패스만 저장 규칙으로 안전하게.
3. **B 단일 창** — (a) 방식이 단순. C·A 이후.

각 항목은 성숙한 시뮬레이터(테스트 340개)를 건드리므로, 하나씩 집중 반복 +
전권 테스트 그린 확인으로 진행. 사용자가 우선순위/동작을 정하면 착수.
